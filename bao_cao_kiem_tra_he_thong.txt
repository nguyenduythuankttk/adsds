[BUG-02] OTP dùng Random.Shared — Không an toàn về mặt cryptographic
────────────────────────────────────────────────────────────────────────────────
  File    : BackEnd/Services/Implementations/AuthService.cs
  Dòng    : 58
  Mức độ  : CRITICAL

  CODE HIỆN TẠI:
    private string GenerateSecureToken()
        => Random.Shared.Next(100000, 999999).ToString();
    //  ↑ Tên phương thức gọi là "Secure" nhưng dùng Random thường

  PHÂN TÍCH:
    - Random.Shared là pseudo-random number generator (PRNG), KHÔNG phải CSPRNG
    - Trong một số điều kiện, output của Random có thể đoán được nếu attacker
      biết seed hoặc quan sát đủ output trước đó
    - OTP này dùng cho: xác thực email, đặt lại mật khẩu, đổi mật khẩu
      → Đây là tính năng bảo mật nhạy cảm cần dùng randomness thật sự an toàn
    - Khoảng giá trị 100000-999999 chỉ có 899,999 giá trị có thể → brute-force
      trong thời gian ngắn nếu không có rate limiting

  ĐỀ XUẤT SỬA:
    using System.Security.Cryptography;

    private string GenerateSecureToken()
        => RandomNumberGenerator.GetInt32(100000, 1000000).ToString();
    //  ↑ RandomNumberGenerator là CSPRNG, an toàn cryptographically

[BUG-07] FEFO vs FIFO không nhất quán — Nguyên liệu sắp hết hạn không được ưu tiên
────────────────────────────────────────────────────────────────────────────────
  File 1  : BackEnd/Services/Implementations/BillService.cs — dòng 545
  File 2  : BackEnd/Services/Implementations/DeliveryInfoService.cs — dòng 45
  Mức độ  : HIGH

  CODE HIỆN TẠI:
    // DineIn (BillService) — FEFO: First Expired First Out ← ĐÚNG cho thực phẩm
    .OrderBy(b => b.Exp)
    .ThenBy(b => b.QuantityOnHand)
    .ThenBy(b => b.ImportDate)

    // Delivery (DeliveryInfoService) — FIFO: First In First Out ← THIẾU FEFO
    .OrderBy(b => b.ImportDate)

  PHÂN TÍCH:
    - FEFO là tiêu chuẩn cho quản lý nguyên liệu thực phẩm (food safety)
    - Delivery order dùng FIFO → nguyên liệu sắp hết hạn không được xuất trước
    - Dễ gây tình trạng nguyên liệu hết hạn phải bỏ (lãng phí chi phí)
    - Hai luồng cùng hệ thống nhưng xử lý khác nhau → inconsistency

  ĐỀ XUẤT SỬA (DeliveryInfoService.cs, dòng 45):
    // Thay:
    .OrderBy(b => b.ImportDate)
    // Thành:
    .OrderBy(b => b.Exp)
    .ThenBy(b => b.ImportDate)

[FE-04] Giá trị QR SePay cho DineIn BankTransfer không bao gồm shipping fee
────────────────────────────────────────────────────────────────────────────────
  File    : FrontEnd/asset/js/api.js — showSePayQrModal, dòng 96

  Mức độ  : MEDIUM (UX issue)

  PHÂN TÍCH:
    - BillService.CreateDeliveryBill: bill.Total = food + VAT (không có phí ship)
    - ShippingFee được lưu riêng trong DeliveryInfo.ShippingFee
    - QR code hiển thị: amount = bill.Total (chỉ food + VAT)
    - Khách hàng quét QR, thấy số tiền KHÔNG bao gồm phí ship
    - DeliveryInfoService khi Delivered check: MoneyReceived >= bill.Total + ShippingFee

  CẦN QUYẾT ĐỊNH NGHIỆP VỤ:
    Option A: BankTransfer trả trước food qua QR, phí ship trả tiền mặt khi nhận
    Option B: QR amount = food + ship + VAT (đồng nhất với Cash flow)

    Nếu chọn Option B:
    // BillService.cs: Thêm shippingFee vào Total trước khi sinh QR
